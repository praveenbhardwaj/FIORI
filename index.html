PlatformBTP Connectivity

BTP Destination Service

A Destination is a named, reusable connection configuration stored in BTP. Instead of hardcoding backend URLs in application code, you reference a destination name — BTP resolves the actual URL, auth method, and proxy settings at runtime.

Northwind OData Destination Setup

FieldValue
NameNorthwind (case-sensitive)
TypeHTTP
URLhttps://services.odata.org
Proxy TypeInternet
AuthenticationNoAuthentication
Additional PropertyWebIDEUsage = odata_gen
⚠️
Case sensitivity: The destination name Northwind must match exactly what is declared in manifest.json under sap.app.dataSources. A mismatch causes a 404 at runtime with no obvious error message.

Referencing a Destination in manifest.json

json — manifest.json
"sap.app": {
  "dataSources": {
    "northwindService": {
      "uri": "/V2/Northwind/Northwind.svc/",
      "type": "OData",
      "settings": { "odataVersion": "2.0" }
    }
  }
}

The URI /V2/Northwind/Northwind.svc/ is appended to the destination's base URL automatically. The full resolved URL becomes https://services.odata.org/V2/Northwind/Northwind.svc/.

ArchitectureSystem Design

Fiori 3-Tier Architecture

Every SAP Fiori application follows a three-tier separation of concerns. Understanding this model is essential for debugging, configuring services, and extending existing apps.

TierTechnologyResponsibility
PresentationSAPUI5 running in the browserRenders UI, handles user interaction, manages data binding
IntegrationSAP Gateway / CAP / BTPExposes business data as OData services, enforces authorisation
BackendSAP S/4HANA, ECC, or customStores and processes actual business data via BAPIs or CDS views
💡
In development with Northwind, the integration tier is replaced by the public Northwind OData service at services.odata.org, removing the need for an SAP backend system.
ArchitectureDesign Pattern

MVC Pattern in SAPUI5

SAPUI5 implements a strict Model-View-Controller (MVC) separation. Each layer has a single, clear responsibility — mixing concerns across layers is a common source of maintenance issues.

LayerFileResponsibilityWhat NOT to put here
View*.view.xmlUI controls, layout, property bindingsBusiness logic, API calls
Controller*.controller.jsEvent handlers, model reads/writes, navigationInline HTML, direct DOM manipulation
ModelDefined in manifest.jsonData store — JSON (local) or OData (remote)UI state logic, navigation

Views & ControllersXML View

XML View Anatomy

In SAPUI5, the preferred view format is XML. XML Views declare the UI control tree declaratively — each XML element maps to a UI5 control class, and attributes map to control properties.

xml
<mvc:View
  controllerName="northwind.controller.Home"
  xmlns:mvc="sap.ui.core.mvc"
  xmlns="sap.m"
  xmlns:l="sap.ui.layout">

  <Page title="{i18n>homeTitle}">
    <headerContent>
      <Button icon="sap-icon://add" press=".onAdd"/>
    </headerContent>
    <content>
      <!-- Controls go here -->
    </content>
  </Page>

</mvc:View>

XML Namespace Declarations

DeclarationEffect
xmlns="sap.m"Default namespace — controls like <Button> resolve to sap.m.Button
xmlns:mvc="sap.ui.core.mvc"Required for the root <mvc:View> element
xmlns:l="sap.ui.layout"Prefix l: for layout controls like <l:HorizontalLayout>
xmlns:f="sap.ui.layout.form"Prefix f: for form controls like <f:SimpleForm>
xmlns:core="sap.ui.core"Prefix core: for Icon, Fragment, and CustomData
⚠️
Every library you use in an XML View must be declared as a namespace. A missing namespace results in a rendering error and no helpful message in the console.
Views & ControllersController

Controller Structure & Lifecycle

A controller is a JavaScript module that handles the event-driven logic for a single view. It follows the AMD module pattern via sap.ui.define.

javascript
sap.ui.define([
  "sap/ui/core/mvc/Controller"
], function(Controller) {
  "use strict";

  return Controller.extend("northwind.controller.Home", {

    onInit: function() {
      // Runs once when view is created
      // Use for: model setup, router listeners, initial data fetch
    },

    onBeforeRendering: function() {
      // Runs before each render cycle — rarely needed
    },

    onAfterRendering: function() {
      // Runs after view is in the DOM — use sparingly
    },

    onExit: function() {
      // Runs on view destruction — clean up timers, listeners
    },

    onButtonPress: function() {
      // Custom handler — referenced in View via press=".onButtonPress"
      sap.m.MessageToast.show("Pressed!");
    }

  });
});

Lifecycle Hook Reference

HookWhen it firesTypical use
onInitOnce, when the view is instantiatedSet models, attach route listeners, load initial data
onBeforeRenderingBefore each renderRarely used; avoid heavy operations
onAfterRenderingAfter view is in the DOMDOM access (avoid if possible — let binding handle updates)
onExitWhen view is destroyedDestroy fragments, clear timers, unsubscribe events
Views & ControllersFragments & Dialogs

Fragments & Dialogs

A Fragment is a reusable XML UI snippet without its own controller. The most common use case is Dialogs — keeping them in separate fragment files keeps views clean and enables reuse.

Fragment File — CreateEmployee.fragment.xml

xml
<core:FragmentDefinition
  xmlns="sap.m"
  xmlns:core="sap.ui.core">

  <Dialog title="Create Employee">
    <content>
      <SimpleForm>
        <Label text="First Name"/>
        <Input value="{newEmp>/firstName}"/>
      </SimpleForm>
    </content>
    <buttons>
      <Button text="Save" type="Emphasized" press=".onSave"/>
      <Button text="Cancel" press=".onClose"/>
    </buttons>
  </Dialog>

</core:FragmentDefinition>

Controller — Load and Manage Dialog

javascript
onOpenDialog: function() {
  if (!this._oDialog) {
    // Load once, cache reference — avoid re-loading on every open
    this._oDialog = sap.ui.xmlfragment(
      "northwind.view.CreateEmployee",
      this   // passes controller as event handler scope
    );
    this.getView().addDependent(this._oDialog);
  }
  this._oDialog.open();
},
onClose: function() { this._oDialog.close(); }
⚠️
Always call addDependent() — this attaches the dialog to the view's lifecycle so it is properly destroyed when the view is destroyed, preventing memory leaks.

Data BindingClient-Side Model

JSON Model

sap.ui.model.json.JSONModel holds data as a plain JavaScript object entirely in the browser. No server calls. Used for UI state, form data, static reference data, and local configuration.

javascript
// Create with initial data
var oModel = new sap.ui.model.json.JSONModel({
  employees: [
    { firstName: "Nancy", title: "Sales Rep" }
  ],
  isLoading: false
});

// Attach to view (local scope)
this.getView().setModel(oModel);

// Attach to component (global scope — all views)
this.getOwnerComponent().setModel(oModel);

// Read a value
var sName = oModel.getProperty("/employees/0/firstName");

// Update a value — UI refreshes automatically
oModel.setProperty("/isLoading", true);

// Replace entire dataset
oModel.setData({ employees: [...] });
💡
Set a named JSONModel for UI state alongside the unnamed OData model: setModel(oUIModel, "ui"). Reference in XML as {ui>/isLoading}. This keeps OData data and UI state cleanly separated.

Data BindingDetail Pattern

Element Binding

Element Binding sets an entire OData entity as the binding context of a view. All child controls within that view can then bind to properties using relative paths — no entity set prefix needed.

javascript — controller
_bindView: function(sEmployeeId) {
  this.getView().bindElement({
    path: "/Employees(" + sEmployeeId + ")",
    events: {
      dataRequested: function() { sap.ui.core.BusyIndicator.show(0); },
      dataReceived: function() { sap.ui.core.BusyIndicator.hide(); }
    }
  });
}
xml — view (relative bindings just work)
<!-- Once bindElement("/Employees(3)") is set, all bindings are relative -->
<Page title="{FirstName} {LastName}">
  <content>
    <SimpleForm>
      <Label text="Title"/><Text text="{Title}"/>
      <Label text="City"/><Text  text="{City}"/>
      <Label text="Phone"/><Text text="{HomePhone}"/>
    </SimpleForm>
  </content>
</Page>

Expression Binding

Expression Binding allows simple conditional logic directly in XML using the {= expression } syntax — without writing a separate formatter function.

xml — common patterns
<!-- Ternary text -->
<Text text="{= ${Status} === 'A' ? 'Active' : 'Inactive' }"/>

<!-- Conditional visibility -->
<Button text="Delete" visible="{= ${role} === 'admin' }"/>

<!-- Arithmetic -->
<Text text="{= 'Total: $' + (${qty} * ${price}) }"/>

<!-- ObjectStatus state from value -->
<ObjectStatus
  state="{= ${UnitPrice} > 20 ? 'Warning' : 'Success' }"
  text="{= ${UnitPrice} > 20 ? 'Premium' : 'Standard' }"/>

RoutingConfiguration

Router Configuration

The SAPUI5 Router manages Single Page Application navigation by watching URL hash changes. All routing configuration lives in manifest.json and the Router is initialised in Component.js.

Core Concepts

Router

Watches hash changes, activates the matching route, and loads the target view.

Route

Maps a URL pattern to a target. Name is used in code; pattern appears in the hash.

Target

Specifies which XML View to load and which App control to render it into.

Pattern

"" = home · about = static · detail/{id} = parameterised · :id: = optional

json — manifest.json routing section
"routing": {
  "config": {
    "routerClass": "sap.m.routing.Router",
    "viewType": "XML",
    "viewPath": "northwind.view",
    "controlId": "app",
    "controlAggregation": "pages",
    "transition": "slide"
  },
  "routes": [
    { "name": "RouteHome",   "pattern": "",                    "target": ["TargetHome"] },
    { "name": "RouteAbout",  "pattern": "about",               "target": ["TargetAbout"] },
    { "name": "RouteDetail", "pattern": "detail/{employeeId}", "target": ["TargetDetail"] }
  ],
  "targets": {
    "TargetHome":   { "viewType": "XML", "viewName": "Home" },
    "TargetAbout":  { "viewType": "XML", "viewName": "About" },
    "TargetDetail": { "viewType": "XML", "viewName": "Detail" }
  }
}

Component.js — Initialise Router

javascript
init: function() {
  // Step 1: call parent — MANDATORY, never skip
  UIComponent.prototype.init.apply(this, arguments);
  // Step 2: start the router — MUST be last
  this.getRouter().initialize();
}
🚫
Calling getRouter().initialize() before UIComponent.prototype.init causes routing to silently fail with no error. Always call parent first.
RoutingSimple Navigation

Navigation Without Parameters

Use navTo(routeName) to navigate to a static view — no data needs to be passed in the URL. The route name must match the "name" field in manifest.json, not the pattern.

javascript
onNavToAbout: function() {
  this.getOwnerComponent().getRouter().navTo("RouteAbout");
}
ℹ️
navTo takes the route name ("RouteAbout"), not the URL pattern ("about"). This is a frequent source of confusion.
RoutingParameterised Navigation

Navigation With Parameters

Pass data in the URL using parameterised route patterns. The parameter key in the navTo call must exactly match the placeholder name in the pattern (e.g. {employeeId}).

Sending — navTo with parameter

javascript — source controller
onSelectEmployee: function(oEvent) {
  var sId = oEvent.getSource()
    .getBindingContext()
    .getProperty("EmployeeID");

  this.getOwnerComponent().getRouter().navTo("RouteDetail", {
    employeeId: sId   // key = placeholder name in manifest pattern
  });
}

Receiving — attachPatternMatched

javascript — destination controller
onInit: function() {
  this.getOwnerComponent().getRouter()
    .getRoute("RouteDetail")
    .attachPatternMatched(this._onMatch, this);
},

_onMatch: function(oEvent) {
  var sId = oEvent.getParameter("arguments").employeeId;
  // sId holds the value from the URL hash
  this._loadEmployee(sId);
},

_loadEmployee: function(sId) {
  this.getOwnerComponent().getModel()
    .read("/Employees(" + sId + ")", {
      success: function(oData) {
        // populate view fields
      }
    });
}

RoutingBack Navigation

Back Navigation

SAPUI5 provides sap.ui.core.routing.History to check whether there is a previous hash to go back to. This enables correct behaviour whether a user navigated into a view normally or arrived via a direct URL/bookmark.

javascript
sap.ui.define([
  "sap/ui/core/mvc/Controller",
  "sap/ui/core/routing/History"
], function(Controller, History) {

  return Controller.extend("northwind.controller.Detail", {

    onNavBack: function() {
      var sPrev = History.getInstance().getPreviousHash();

      if (sPrev !== undefined) {
        // Normal back — use browser history
        window.history.go(-1);
      } else {
        // No history (direct URL) — navigate to home
        this.getOwnerComponent().getRouter().navTo("RouteHome");
      }
    }

  });
});
ODataFundamentals

OData — Concepts & Terminology

OData (Open Data Protocol) is a REST-based standard for exposing data via HTTP. SAP Gateway and CAP expose SAP business data as OData services, which SAPUI5 consumes via its built-in ODataModel.

TermMeaningExample
Service RootBase URL of the OData servicehttps://services.odata.org/V2/Northwind/Northwind.svc/
EntitySetA collection of records — like a table/Employees, /Products
EntityTypeThe schema of a single recordEmployee with properties FirstName, LastName, etc.
KeyUnique identifier for a single record/Employees(3) — EmployeeID = 3
$metadataMachine-readable service schemaAppend /$metadata to service root URL
$filterServer-side filtering via URL query param/Employees?$filter=City eq 'London'
$expandInclude related entities in response/Employees?$expand=Orders
$top / $skipPagination?$top=10&$skip=20
ODataRead Operations

Read Operations

Data can be read through automatic binding (preferred for lists) or via the explicit model.read() method (used when you need precise control over fetch timing or need to apply filters programmatically).

javascript
var oModel = this.getOwnerComponent().getModel();

// Read a single entity by key
oModel.read("/Employees(1)", {
  success: function(oData) { console.log(oData.FirstName); },
  error:   function(oErr)  { sap.m.MessageToast.show("Error"); }
});

// Read with server-side filter
var oFilter = new sap.ui.model.Filter(
  "City",
  sap.ui.model.FilterOperator.EQ,
  "London"
);
oModel.read("/Employees", {
  filters: [oFilter],
  success: function(oData) { console.log(oData.results); }
});

// Read with $expand — include related Orders
oModel.read("/Employees(1)", {
  urlParameters: { "$expand": "Orders" },
  success: function(oData) { console.log(oData.Orders); }
});
ODataWrite Operations

Create, Update & Delete

The OData v2 model exposes create(), update(), and remove() for write operations. Each accepts a path, an optional payload, and success/error callbacks.

Create

javascript
oModel.create("/Employees", {
  FirstName: "John", LastName: "Smith", Title: "Developer"
}, {
  success: function(oData) { sap.m.MessageToast.show("Created: " + oData.EmployeeID); },
  error:   function()      { sap.m.MessageToast.show("Create failed"); }
});

Update (PATCH / MERGE)

javascript
oModel.update("/Employees(3)", { Title: "Senior Developer" }, {
  success: function() { sap.m.MessageToast.show("Updated"); oModel.refresh(); },
  error:   function() { sap.m.MessageToast.show("Update failed"); }
});

Delete

javascript
sap.m.MessageBox.confirm("Delete this employee?", {
  onClose: function(sAction) {
    if (sAction === sap.m.MessageBox.Action.OK) {
      oModel.remove("/Employees(3)", {
        success: function() { sap.m.MessageToast.show("Deleted"); },
        error:   function() { sap.m.MessageToast.show("Delete failed"); }
      });
    }
  }
});
OperationMethodHTTP VerbNotes
Createmodel.create()POSTPass entity set path + payload object
Update (partial)model.update()MERGE / PATCHDefault — sends only changed fields
Update (full)model.update({merge:false})PUTReplaces entire entity — use carefully
Deletemodel.remove()DELETEPass entity path with key
💡
Always confirm before delete using MessageBox.confirm(). This is a mandatory SAP Fiori Design Guideline requirement — Fiori apps must not perform destructive actions without user confirmation.
⚠️
The public Northwind service at services.odata.org is read-only. Create, update, and delete operations will return 4xx errors. In production, the backend OData service must explicitly enable write operations.
Smart Controlssap.ui.comp

SmartTable & SmartFilterBar

Smart Controls are metadata-driven — they read the OData service's $metadata and automatically generate columns, filter fields, and labels. Zero manual column configuration required.

When to Use Smart Controls

  • When your OData service is well-annotated with CDS annotations or OData V4 vocabularies
  • When you need built-in export to Excel, personalisation, and sort/filter without extra code
  • In enterprise apps where consistent metadata-driven UI is preferred over custom layouts
⚠️
Smart Controls require a real OData service with valid $metadata. They do not work with JSONModel. Additionally, they belong to the sap.ui.comp library which must be declared as a dependency in manifest.json.

Setup

xml
<mvc:View
  xmlns:smartTable="sap.ui.comp.smarttable"
  xmlns:smartFilterBar="sap.ui.comp.smartfilterbar"
  xmlns:mvc="sap.ui.core.mvc"
  xmlns="sap.m">

  <smartFilterBar:SmartFilterBar
    id="filterBar"
    entityType="Employee"
    search=".onSearch"/>

  <smartTable:SmartTable
    id="smartTable"
    entitySet="Employees"
    smartFilterId="filterBar"
    tableType="ResponsiveTable"
    enableAutoBinding="true"/>

</mvc:View>
UI PatternsDisplay Logic

Custom Formatters

A formatter is a pure JavaScript function that transforms a raw model value into a display-ready string or type. Formatters are kept in webapp/model/formatter.js and referenced in XML view bindings.

javascript — webapp/model/formatter.js
sap.ui.define([], function() {
  "use strict";
  return {

    statusText: function(sStatus) {
      var map = { "A": "Active", "I": "Inactive", "P": "Pending" };
      return map[sStatus] || "Unknown";
    },

    statusState: function(sStatus) {
      return sStatus === "A" ? "Success" : "Error";
    },

    currency: function(nValue, sCurrency) {
      if (!nValue) return "—";
      return (sCurrency || "$") + " " + parseFloat(nValue).toFixed(2);
    }

  };
});

Import and Use in Controller

javascript — controller
sap.ui.define([
  "sap/ui/core/mvc/Controller",
  "northwind/model/formatter"
], function(Controller, formatter) {
  return Controller.extend("northwind.controller.Home", {
    formatter: formatter  // expose to XML View
  });
});

Reference in XML View

xml
<ObjectStatus
  text="{path: 'Status', formatter: '.formatter.statusText'}"
  state="{path: 'Status', formatter: '.formatter.statusState'}"/>

<Text text="{parts: ['UnitPrice','Currency'], formatter: '.formatter.currency'}"/>

Internationalisation (i18n)

All user-visible text in a Fiori app must be externalised into i18n.properties files. This is not just a best practice — it is a mandatory SAP Fiori UX requirement. Hardcoded strings in XML views will fail Fiori app validation.

i18n.properties

properties
# App
appTitle=Northwind Employee App

# Views
homeTitle=Employee List
detailTitle=Employee Detail

# Buttons
createBtn=Create Employee
saveBtn=Save
cancelBtn=Cancel
deleteBtn=Delete

# Messages
deleteConfirm=Are you sure you want to delete this employee?
saveSuccess=Employee saved successfully.

Use in XML View

xml
<Page title="{i18n>homeTitle}">
  <Button text="{i18n>createBtn}" press=".onOpen"/>
</Page>

Use in Controller

javascript
var sMsg = this.getView().getModel("i18n")
  .getResourceBundle()
  .getText("saveSuccess");
sap.m.MessageToast.show(sMsg);

Glossary

Definitions for all key terms across SAP BTP, Fiori, SAPUI5, OData, and routing.

BTP BTP
SAP Business Technology Platform — SAP's unified cloud platform. Hosts Cloud Foundry apps, HANA databases, Integration Suite, and all services consumed by Fiori applications.
Cloud Foundry BTP
The PaaS runtime inside BTP. Handles server infrastructure, auto-scaling, and app lifecycle. SAPUI5 apps are deployed here as MTA archives.
BAS BTP
Business Application Studio — browser-based cloud IDE for SAP development. Replaces WebIDE. Accessed from the BTP cockpit; no local install needed.
Dev Space BTP
An isolated containerised environment inside BAS. Each space is pre-loaded with tools for a specific development type (e.g. SAP Fiori).
Destination BTP
A named HTTP connection stored in BTP. Holds base URL, auth type, and proxy settings. Referenced by name in manifest.json so apps never hardcode backend URLs.
MTA BTP
Multi-Target Application — a deployable archive (.mtar) packaging all modules and service bindings. Described by mta.yaml.
mta.yaml BTP
Deployment descriptor defining all modules (app code) and resources (BTP services) for a deployment to Cloud Foundry.
HTML5 Apps Repo BTP
BTP managed service that hosts and serves SAPUI5 static files. All deployed Fiori apps are stored and served from here.
CF CLI BTP
Cloud Foundry Command Line Interface. Used to login (cf login), deploy (cf deploy), and manage apps. Pre-installed in BAS terminals.
Subaccount BTP
An isolated environment within a BTP Global Account. Has its own CF space, service entitlements, and user roles. Trial accounts include one subaccount.
SAP Fiori Fiori
SAP's UX design system and application framework. Defines design guidelines, interaction patterns, and UI standards. SAPUI5 is the technical foundation.
FLP Fiori
Fiori Launchpad — the browser-based home screen showing app tiles. Provides the outer shell, header, notifications, and cross-app navigation.
Shell Fiori
The outer container provided by FLP. Contains header bar, notifications, and user menu. All Fiori apps run inside the shell.
Semantic Object Fiori
A named business entity (e.g. "Employee") used for intent-based navigation in FLP. Combined with an action (e.g. "display") it forms a navigation target.
sap_horizon Fiori
The latest SAP Fiori design theme. Set via data-sap-ui-theme="sap_horizon" in index.html. Provides the modern Fiori 3 visual language.
SAPUI5 SAPUI5
SAP's JavaScript UI framework. Implements MVC, data binding, and a large library of enterprise controls. Powers all Fiori apps.
Component.js SAPUI5
App entry point. Extends UIComponent. Initialises the router and optionally sets up global models. metadata.manifest:"json" is the standard pattern.
manifest.json SAPUI5
App descriptor — defines app ID, data sources, models, routing, and dependencies. The most important file in a SAPUI5 project.
sap.m SAPUI5
The core mobile-optimised control library. Contains Page, Button, Table, List, Input, Dialog, etc. The default library in Fiori apps.
sap.ui.comp SAPUI5
Smart Controls library — SmartTable, SmartFilterBar, SmartForm. Metadata-driven controls that auto-configure from OData annotations.
onInit SAPUI5
Controller lifecycle hook. Fires once when the view is created. The standard place for model setup, router listener registration, and initial data loading.
Fragment SAPUI5
A reusable XML UI snippet without a controller. Typically used for dialogs. Loaded via sap.ui.xmlfragment().
Formatter SAPUI5
A JS function that transforms a model value for display. Defined in formatter.js, referenced in XML as formatter: '.formatter.myFn'.
Expression Binding SAPUI5
Inline logic in XML using {= expression } syntax. Supports ternary, arithmetic, and comparisons without a separate formatter file.
i18n SAPUI5
Internationalisation model. Text stored as key=value in i18n.properties. Referenced in XML as {i18n>keyName}. Mandatory per Fiori guidelines.
MessageToast SAPUI5
Non-blocking brief notification at screen bottom. Use to confirm actions: sap.m.MessageToast.show("Saved!").
MessageBox SAPUI5
Modal confirmation or error dialog. Types: alert, confirm, error, warning. Required before destructive actions per Fiori guidelines.
addDependent() SAPUI5
Attaches a fragment/dialog to the view lifecycle. Ensures proper cleanup when the view is destroyed; prevents memory leaks.
Demo Kit SAPUI5
Official SAPUI5 documentation portal at ui5.sap.com. Contains API reference, live samples, icon explorer, and design guidelines.
growing="true" SAPUI5
Table/List property enabling lazy loading — only a threshold of rows load initially, more appear on scroll. Essential for large datasets.
OData OData
Open Data Protocol — REST standard for exposing data via HTTP. SAP Gateway exposes SAP data as OData. SAPUI5 has a built-in ODataModel.
EntitySet OData
A collection of entities, analogous to a database table. Accessed at e.g. /Employees.
EntityType OData
The type definition (schema) for a single entity — its property names and data types. Smart Controls read EntityType from $metadata.
$metadata OData
Machine-readable schema of the OData service. Access by appending /$metadata to the service URL. Describes all EntityTypes, EntitySets, and relationships.
model.read() OData
Explicit data fetch from an OData path. Returns data in success callback. Used when binding alone is insufficient.
model.create() OData
HTTP POST to create a new entity. Takes entity set path, payload, and callbacks.
model.update() OData
HTTP MERGE/PATCH to update an existing entity. Sends only the fields provided in the payload by default.
model.remove() OData
HTTP DELETE for a specific entity path (with key). Always confirm with the user before calling.
Northwind OData
Public sample OData service at services.odata.org. Contains Employees, Products, Orders, Customers. Read-only. Used as practice backend in development.
$filter OData
URL query parameter for server-side filtering. E.g. ?$filter=City eq 'London'. Passed as filters array in model.read() calls.
$expand OData
URL query parameter to include related entity data in a single request. E.g. ?$expand=Orders includes Order records with each Employee.
Router Routing
SAPUI5 component that watches URL hash changes and activates the matching route. Initialised via this.getRouter().initialize() in Component.js.
Route Routing
A named URL mapping rule in manifest.json routing config. Defines a URL pattern and which target to activate when the pattern matches.
Target Routing
Specifies which XML View to load and which App control to render it into. Activated by a Route match.
navTo() Routing
Programmatic navigation method. Takes route name and optional params: router.navTo("RouteDetail", {id: 1}).
attachPatternMatched Routing
Registers a handler that fires each time a specific route's pattern is matched. Used in the destination controller to extract parameters and load data.
History Routing
sap.ui.core.routing.History — tracks browser navigation history. Used in back navigation: History.getInstance().getPreviousHash().
URL Hash Routing
The fragment after # in the URL (e.g. #/detail/3). SAPUI5 Router reacts to hash changes for SPA navigation — no page reload occurs.
SPA Routing
Single Page Application — the entire app runs in one HTML page. JavaScript swaps view content in response to navigation events, without full page reloads.
